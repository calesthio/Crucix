# Crucix dev cycle 003

- Date: 2026-04-23
- Objective: reduce OpenSky 429 pressure by adding query rotation and cooldown behavior so cached fallback can become useful in live operation
- Status: completed

## Planned work

1. Reduce the number of OpenSky hotspot queries per sweep.
2. Persist per-source runtime state for hotspot rotation and 429 cooldown.
3. Preserve full hotspot output shape by carrying forward cached results when a region is not queried in the current rotation.
4. Restart the local runtime and verify live health/data behavior.

## Results

- Added OpenSky runtime state file at `runs/cache/opensky-state.json`.
- OpenSky now queries a rotating batch of 4 hotspot regions per sweep instead of all 10 every time.
- Added 30 minute cooldown behavior after HTTP 429 responses.
- Added carry-forward behavior so non-queried regions can use cached hotspot results instead of pretending they were freshly checked.
- Preserved full output shape with explicit stale placeholders when a region was not queried and no cache exists yet.
- Restarted the local Crucix runtime and verified live operation.

## Evidence

- Direct source validation:
  - `node apis/sources/opensky.mjs`
  - returned `queryMode: "rotating-batch"`
  - returned `queriedRegions: ["Middle East", "Taiwan Strait", "Ukraine Region", "Baltic Region"]`
- Runtime state persistence:
  - `runs/cache/opensky-state.json` showed `cursor: 4`
- Live runtime after restart:
  - `curl -fsS http://127.0.0.1:3117/api/health`
  - returned `sourceHealthSummary` with `ok: 24`, `failed: 4`
- Live data after restart:
  - `/api/data` showed OpenSky normalized source health as:
    - `state: "ok"`
    - `error: null`
  - `airMeta.liveTotal` was `1175`

## Notes

- This cycle materially improves the chance of OpenSky earning real cached snapshots for later degraded fallback use.
- The immediate live validation after deploy showed a healthy OpenSky fetch instead of the repeated zero-air / 429-only pattern seen earlier.
