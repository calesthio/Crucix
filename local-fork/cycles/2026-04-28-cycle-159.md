# Crucix Dev Cycle 159 - Added local-admin nonce protection for settings writes

- Date: 2026-04-28
- Plan item: `epic-36 / effort-36.2`
- Status: completed

## Goal
Add an explicit local-admin nonce, CSRF-style write confirmation boundary for browser-originated settings writes so `/api/settings/operator` and `/api/settings/import` are not relying on local-network reachability alone.

## Changes shipped
### Server-side local write-auth contract
- Added a process-scoped local admin write nonce and published it only through the local admin contract.
- Added a versioned write-auth contract that advertises:
  - required status
  - request header name
  - JSON body field
  - active token for the current runtime
- Enforced this nonce on:
  - `PUT /api/settings/operator`
  - `POST /api/settings/import`
- Added bounded error responses for missing or invalid tokens:
  - `428 local-admin-write-token-required`
  - `403 local-admin-write-token-invalid`

### Admin surface integration
- Updated `dashboard/public/admin-settings.html` so the admin surface caches the active write-auth contract from `/api/settings/admin` and sends the nonce in both header and body on save/import.
- Extended the admin controls card to show the write-auth header contract and whether the nonce boundary is active.

### Contract coverage
- Extended persistence and endpoint tests so admin contracts now assert:
  - `admin.controls.writeAuthHeader`
  - `admin.controls.writeAuthBodyField`
  - `admin.writeAuth.required`
  - `admin.writeAuth.token`
- Added negative-path coverage that verifies settings writes fail with `428` when the nonce is omitted, even if the revision token is correct.

## Notes
- The nonce is process-scoped, so a local runtime restart naturally rotates it.
- This is not a replacement for the existing revision/ETag concurrency protection. The two controls now serve different purposes: nonce for write authorization from local browser surfaces, revision token for stale-write conflict detection.

## Runtime rollout note
- Rolling the local runtime via `POST /api/runtime/control` with `restart-safe` exposed a real helper portability issue: the retained restart audit recorded `spawn lsof ENOENT`.
- Crucix still came back locally on `:3117` under PID `76389`, and the live admin/data contracts verified the new write-auth fields after the runtime rotated.
- Captured the helper portability gap in the backlog instead of silently ignoring it.
