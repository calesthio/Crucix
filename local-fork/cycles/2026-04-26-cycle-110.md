# Crucix Dev Cycle 110 - Cheap provider readiness checks and failure classification

- Date: 2026-04-26
- Plan item: `epic-31 / effort-31.7`
- Status: completed

## Goal
Add provider-specific cheap readiness checks and richer failure classification for LLM probes so the readiness heartbeat can distinguish network failure, auth failure, rate limiting, and missing-model states without relying only on a synthetic completion.

## Changes shipped
- Added shared probe-failure classification helpers in `lib/llm/provider.mjs` so readiness errors now carry structured metadata:
  - `classification`
  - `status`
  - `retryable`
  - provider name
- Kept the base provider probe as a bounded synthetic completion fallback, but made it emit structured probe metadata on failure.
- Added provider-specific cheap probe paths that avoid a synthetic completion when possible:
  - `ollama` uses `/api/tags`
  - `openai`, `grok`, `mistral`, and `minimax` use model metadata lookups
  - `openrouter` uses the models catalog
  - `gemini` uses model metadata lookup
- Extended readiness heartbeat state in `server.mjs` with:
  - `lastProbeType`
  - `lastFailure.classification`
  - `lastFailure.status`
  - `lastFailure.retryable`
- Updated readiness summaries so `/api/health` and `/api/llm/operations` now distinguish:
  - network failure
  - timeout
  - auth rejection
  - rate limiting
  - missing model
  - provider-side errors
  - request errors
- Updated `dashboard/public/llm-ops.html` to render operator-visible `Probe type`, `Failure class`, and `Retryable` fields.
- Added regression coverage in `test/llm-provider-probe.test.mjs` and expanded settings/endpoint contract assertions.
