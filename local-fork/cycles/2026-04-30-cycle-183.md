## Goal
Separate raw sweep completion from actual snapshot publish success, and tighten startup-sweep tracing so health does not imply a fresh published dashboard when the runtime is still stuck in synthesis.

## Backlog review and promotions
- Reviewed the backlog and promoted `backlog-64-a` into the active roadmap as `epic-64 / effort-64.1`.
- Added follow-on backlog item `backlog-64-b` to expose sweep-phase timing more directly on `/api/health` and reconcile the current top-level `runtimePhase` gap.
- Marked the prior retained-state investigation slice `epic-63 / effort-63.1` completed, because cycle 182 narrowed the issue to startup synthesis/publish rather than a post-publish gate leak.

## Changes shipped
- Reclassified `lastSweepTime` in `server.mjs` to represent the last successfully published snapshot rather than raw sweep completion.
- Added `lastBriefingCompletedAt` and `lastRawSnapshotPersistedAt` runtime state so raw-source completion and `runs/latest.json` persistence are tracked separately from dashboard publish success.
- Extended `runtimeControl.lastSuccess` to expose:
  - `publishedAt`
  - `rawSweepCompletedAt`
  - `rawSnapshotPersistedAt`
- Delayed `lastSweepTime` assignment until after `currentData` is updated and the snapshot broadcast is emitted, so `/api/health` no longer claims a newer published sweep while `/api/data` is still stale or absent.
- Added a regression in `test/sweep-lifecycle-transitions.test.mjs` proving `runSweepCycle()` keeps the old published timestamp until synthesis completes and the snapshot is actually published.
- Advanced `local-fork/plan.json` cycle state to 183 and completed `epic-64 / effort-64.1`.
