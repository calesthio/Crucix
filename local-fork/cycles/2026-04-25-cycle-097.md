# Crucix Dev Cycle 097 - Runtime control surface and last-success state

- Date: 2026-04-25
- Plan item: `epic-29 / effort-29.3`
- Status: completed

## Goal
Expose runtime process controls and last-success state so operators can see current sweep stage, active PID identity, last good refresh state, and bounded local restart/stop actions instead of inferring runtime posture from scattered health fields.

## Changes shipped
- Added a versioned runtime control contract via `buildRuntimeControlContract()`:
  - `version: runtime-control-v1`
  - live process identity (`pid`, `port`, `cwd`, `startedAt`, `uptimeSeconds`)
  - current sweep phase and watchdog metadata
  - last completed, failed, and recovered phase breadcrumbs
  - last-success summary (`publishedAt`, snapshot timestamp, source counts, ideas source, agent-analysis state)
  - bounded local action contract with allowed actions and endpoint metadata
- Extended `/api/health` with:
  - `runtimeControl`
  - `lastSuccess`
- Extended admin settings contract with:
  - `runtimeControl`
  - `admin.controls.runtimeControlEndpoint`
- Added local-only runtime control endpoint:
  - `POST /api/runtime/control`
  - allowed actions:
    - `restart-safe`
    - `stop`
  - unsupported actions return `400`
- Added admin surface controls in `dashboard/public/admin-settings.html`:
  - Restart button
  - Stop button
  - runtime control card showing PID, port, current phase, last success, and allowed actions
- Added `logs/runtime-control.log` append target for queued control actions.
