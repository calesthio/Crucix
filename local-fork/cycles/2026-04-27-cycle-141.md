# Crucix Dev Cycle 141 - Added urgent critical-event candidate queue and promoted backlog into roadmap

- Date: 2026-04-27
- Plan item: `epic-35 / effort-35.2`
- Status: completed

## Goal
Build the first urgent critical-event candidate queue so low-trust and medium-trust tripwire signals can persist in monitoring state across sweeps, retry corroboration on a short cadence, and promote or discard based on the class policy thresholds defined in cycle 140.

## Changes shipped
- Promoted the unscheduled local runtime restart-verification backlog item into `local-fork/plan.json` as `epic-37 / effort-37.1`.
- Added a persisted critical-event queue state keyed as `critical-events:queue`.
- Added first-pass candidate classification for the existing critical-event taxonomy:
  - `governmentSiteViolence`
  - `aviationIncident`
  - `radiationAnomaly`
  - `chokepointDisruption`
- Built queue processing that:
  - ingests current suspect and corroborated signals
  - classifies matching signals into urgent candidates
  - records trust-tier counts and official-confirmation hits
  - retains monitoring candidates across sweeps with retry scheduling
  - promotes candidates when configured corroboration thresholds are met
  - discards stale or exhausted candidates after bounded retries
- Exposed the queue contract as `critical-event-queue-v1` on:
  - `GET /api/data` as `criticalEventQueue`
  - `GET /api/health` as `criticalEventQueue`
  - `GET /api/settings` under `alerts.criticalEvents.queue`
- Updated contract tests to validate queue exposure on live UI-facing and operator-facing surfaces.
- Updated `local-fork/plan.json` to mark `effort-35.2` complete, promote the restart verification roadmap item, advance cycle tracking to 141, and record a new backlog recommendation for classifier hardening.
