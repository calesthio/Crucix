# Crucix Dev Cycle 186 - Published versus candidate snapshot timestamp honesty

- Date: 2026-04-30
- Plan item: `epic-67 / effort-67.1`
- Status: completed with follow-on noted

## Goal
Promote the pending backlog recommendation into the roadmap, then split published snapshot timestamp from the in-flight candidate snapshot timestamp so `/api/health` stops conflating stale published dashboard state with work still in progress.

## Backlog review and promotions
- Promoted `backlog-66-a` into the active roadmap as `epic-67 / effort-67.1`.
- Added follow-on `backlog-67-a` after live validation showed `candidateSnapshotTimestamp` can still remain null when the raw sweep payload itself lacks `meta.timestamp`.

## Changes shipped
- Added new runtime state in `server.mjs`:
  - `lastPublishedSnapshotTimestamp`
  - `candidateSnapshotTimestamp`
- Updated `buildRuntimeControlContract()` so `lastSuccess` now exposes:
  - `publishedSnapshotTimestamp`
  - `candidateSnapshotTimestamp`
  - compatibility alias `snapshotTimestamp`, now explicitly tied to the published snapshot timestamp
- Set `lastPublishedSnapshotTimestamp` when:
  - previously persisted `runs/latest.json` is loaded at startup
  - a new synthesized snapshot is actually published
- Set and clear `candidateSnapshotTimestamp` around the active sweep lifecycle so unpublished work is tracked separately from the published snapshot.
- Extended `test/sweep-lifecycle-transitions.test.mjs` to prove the old published snapshot timestamp is retained while a new candidate is still pending, and only updates after publish succeeds.
- Extended `test/settings-endpoint-contract.test.mjs` to assert the new health-contract fields exist and that `snapshotTimestamp` matches `publishedSnapshotTimestamp`.
- Advanced `local-fork/plan.json` cycle state to 186 and completed `epic-67 / effort-67.1`.

## Validation
Executed:
- `node --test --test-concurrency=1 test/sweep-lifecycle-transitions.test.mjs test/settings-endpoint-contract.test.mjs`
- `npm run restart:safe`
- repeated live checks against `http://127.0.0.1:3117/api/health`

Results:

  - `livePid: 89998`
  - `rotationProved: true`
  - `healthStatus: ok`
- Live startup health now separates published snapshot timing cleanly:
  - briefing samples showed `publishedSnapshotTimestamp: 2026-04-30T13:24:00.620Z`
  - `publishedAt` stayed `null` for the new process while the old published snapshot timestamp remained visible as prior published state
  - `snapshotTimestamp` matched `publishedSnapshotTimestamp`
  - synthesis samples showed `rawSweepCompletedAt: 2026-04-30T13:30:04.349Z` without overwriting the published snapshot timestamp
- Follow-on gap observed live:
  - `candidateSnapshotTimestamp` remained `null` even once the sweep reached synthesis
  - that indicates some raw sweep payloads do not provide `meta.timestamp`, so the candidate path still needs its own explicit timestamp source

## Follow-on recommendation added to backlog
- `backlog-67-a` Ensure `candidateSnapshotTimestamp` is populated even when the raw sweep payload lacks `meta.timestamp`, either by stamping a candidate timestamp at briefing completion or by exposing a separate candidate-generation timestamp.
