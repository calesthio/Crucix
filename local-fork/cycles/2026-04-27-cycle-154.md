# Crucix Dev Cycle 154 - Clamped the dashboard map back to sane proportions

- Date: 2026-04-27
- Plan item: `epic-45 / effort-45.1`
- Status: completed

## Goal
Evaluate the live dashboard map window size, confirm whether it had grown disproportionate to the visible dashboard window, and ship a sizing fix that keeps the map reasonably bounded by layout mode.

## Findings
- Live measurement on the running dashboard at `http://127.0.0.1:3117/` in a 1280x720 browser session showed the map container rendering at roughly `656x1053` in desktop layout.
- That meant the map panel was taller than the entire visible browser window, which pushed the lower center-column content far below the fold and made the map dominate the dashboard disproportionately.
- The practical cause was that the map container was allowed to grow as a flex item with a large minimum height and no viewport-aware upper bound.

## Changes shipped
- Promoted dashboard map proportional sizing into `local-fork/plan.json` as `epic-45 / effort-45.1` and completed it in the same cycle.
- Updated `dashboard/public/jarvis.html` CSS for `.map-container`:
  - replaced the unbounded `flex:1; min-height:560px` behavior with explicit height clamps
  - default / desktop map sizing now uses `height: clamp(420px, 58vh, 680px)`
  - narrow layout now uses `height: clamp(320px, 50vh, 460px)`
  - wallboard layout now uses `height: clamp(520px, 68vh, 860px)`
  - set `flex: 0 0 auto` so the map stops freelancing vertically against the rest of the center column
- Left the map width responsive while capping vertical growth per layout mode.
- Advanced cycle tracking to 154.

## Validation
- Live browser re-measurement after the CSS fix on the running dashboard showed:
  - viewport: `1280x720`
  - map before fix: about `656x1053`
  - map after fix: about `656x420`
  - layout mode remained `desktop`
- The updated live dashboard served the fixed CSS immediately from the running runtime.
- Saved before/after inspection artifacts:
  - `local-fork/artifacts/dashboard-map-current.png`
  - `local-fork/artifacts/dashboard-map-fixed.png`

## Backlog and follow-up
- Added `backlog-2026-04-27-dashboard-map-height-tokens` so map height clamps can move into explicit layout tokens or a shared sizing helper instead of remaining scattered inline in CSS.

## Notes
- This was a real live-layout bug, not a cosmetic preference tweak. The map had grown taller than the viewport, which broke the dashboard’s overall visual budget and made the center column feel hijacked by the globe.
