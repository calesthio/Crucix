# Crucix Dev Cycle 146 - Hardened critical-event classification with richer metadata cues

- Date: 2026-04-27
- Plan item: `epic-39 / effort-39.1`
- Status: completed

## Goal
Reduce false negatives and alias-driven brittleness in the critical-event queue by moving beyond regex on headline text alone and incorporating richer source, topic, and metadata cues into class detection.

## Changes shipped
- Added `buildCriticalEventClassificationText(signal)` in `server.mjs` to aggregate a broader classification corpus from:
  - `signal`, `reason`, `label`, `region`
  - `evidenceSource`, `sourceName`, `sourceId`
  - `category`, `type`, `kind`, `topic`
  - `topics`, `tags`, `keywords`, `entities`
  - nested `metadata` fields including topic/category/type/source/tags/topics
- Reworked `classifyCriticalEventSignal(signal)` from a simple first-match regex chain into a lightweight cue-scoring model across:
  - `radiationAnomaly`
  - `aviationIncident`
  - `governmentSiteViolence`
  - `chokepointDisruption`
- Added negative-context downgrades for phrases like `no incident reported`, `exercise only`, `drill only`, `routine policy update`, and `funding announcement` so weak keyword overlap does not create spurious classifications.
- Added regression coverage proving:
  - government-site violence can be inferred from richer metadata such as `secret service` and `white house`
  - aviation incidents can be inferred from topic and source metadata such as `aviation operations` and `opensky-network`
  - weak ambiguous wording is discarded instead of forced into a class
- Marked `effort-39.1` complete and advanced local cycle tracking to 146.

## Notes
- This is still a bounded heuristic classifier, not a learned model, but it is materially less fragile than the earlier regex-only first-match path.
- The new ambiguity handling is deliberate: when the signal only weakly overlaps multiple classes, Crucix now prefers no classification over a confident lie.
