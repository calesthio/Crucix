# Crucix Dev Cycle 189 - Runtime history cache reuse for sweep summaries

- Date: 2026-05-01
- Plan item: `epic-70 / effort-70.1`
- Status: completed locally, remote push blocked by repository permissions

## Goal
Reduce avoidable memory churn and peak heap pressure by reusing the reconstructed persisted runtime history during sweep summary generation instead of repeatedly rebuilding the full SQLite and cold-history corpus in one pass.

## Backlog review and promotions
- Added a new direct roadmap slice as `epic-70 / effort-70.1` for runtime history cache reuse across sweep summary builders.
- No separate backlog promotion was needed because this became the next direct work slice by request after live investigation.

## Findings
- Live Crucix was not in an active watchdog recovery loop during investigation.
- Historical resets were consistent with local `restart-safe` and explicit restart activity under launchd `KeepAlive`, not a confirmed spontaneous crash loop.
- A real memory-growth contributor existed in the repo: persisted runtime history was being fully reloaded, normalized, and sorted repeatedly across trend/history builders even though those calls happen in the same sweep path.

## Changes shipped
- Added `allRunsCache` in `lib/delta/memory.mjs` so `_getAllRunsNewestFirst()` reuses the reconstructed full-history view within the process instead of re-reading SQLite and cold JSON on every history/trend request.
- Added targeted cache invalidation when a run is added or archived, preserving correctness while avoiding repeated full-history rebuilds.
- Extended `test/runtime-history-store.test.mjs` to verify repeated history/trend readers reuse one persisted history load on a fresh `MemoryManager` instance.
- Updated `test/trend-history-backfill.test.mjs` to use relative timestamps so the legacy-backfill regression remains valid independent of wall-clock date.
- Committed the fix locally as `b77170f` (`Cache reconstructed runtime history across sweep summaries`).
