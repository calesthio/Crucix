# Crucix dev cycle 087 - operator settings control plane

## Plan target
- Epic 27, effort 27.1
- Create a dedicated settings surface that centralizes layout preferences, source selection, LLM provider/model controls, agent-analysis behavior, refresh cadence, and debug exposure into one operator-facing control plane.

## What changed
- Added a versioned operator settings contract in `server.mjs` exposed at `/api/settings` as `operator-settings-v1`.
- Centralized current layout posture, source inventory and selection summary, LLM provider and runtime state, agent-analysis posture, refresh cadence, debug exposure, and alert-routing status into one read-first control-plane payload.
- Added a dedicated `/settings` UI shell in `dashboard/public/settings.html` that renders the control-plane sections for operators.
- Added a Settings launcher button to the main dashboard top bar and injected `settingsUrl` into runtime HTML bootstrap data.
- Reused truthful runtime values instead of inventing write-capable settings that do not exist yet.

  - `/api/settings` returned `version: operator-settings-v1`, `layout.current: default-terminal`, `llm.provider: ollama`, and all expected sections.
  - `/settings` served the UI shell containing `Settings control plane`, a Dashboard link, and raw `/api/settings` access.
