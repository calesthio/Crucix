# Crucix Cycle 051 - 2026-04-24

## Summary
- Completed Epic 15 effort 15.2: add a sweep watchdog so a hung in-progress gate can self-heal and expose explicit operator telemetry.
- This cycle did not try to kill or cancel the underlying async work. Instead, it recovers the runtime gate honestly so the scheduler and operator surfaces stop being trapped behind a stale `sweepInProgress` state.

## What changed
- Added configurable watchdog settings in `crucix.config.mjs`:
  - `SWEEP_WATCHDOG_TIMEOUT_MINUTES`
  - `SWEEP_WATCHDOG_POLL_SECONDS`
- Added watchdog state and helpers in `server.mjs`:
  - `getSweepWatchdogSnapshot()`
  - `recoverHungSweep()`
  - `runSweepWatchdog()`
- Added `/api/health.sweepWatchdog` with:
  - timeout/poll configuration
  - active/overdue state
  - overdue age
  - recovery telemetry (`recoveryCount`, `lastRecoveryAt`, `lastRecoveryReason`, `lastRecoveredSweepStartedAt`)
- If `runSweepCycle()` sees `sweepInProgress`, it now asks the watchdog whether the gate is stale before blindly skipping.
- Added a recurring watchdog interval so overdue gates can be cleared even without a manual or scheduled sweep attempt.
- Added regression coverage in `test/sweep-watchdog.test.mjs` for both overdue recovery and healthy no-op behavior.

- Controlled runtime validation with `SWEEP_WATCHDOG_TIMEOUT_MINUTES=1` confirmed the safety floor clamps to a 5 minute minimum timeout rather than allowing an unrealistically aggressive auto-recovery threshold.
