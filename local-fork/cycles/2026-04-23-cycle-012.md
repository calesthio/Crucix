- Objective: make chat context-state handling explicit and debuggable for iMessage-style follow-ups, and update backlog as needed
- Status: completed

## Planned work

1. Add any new unscheduled recommendations to the backlog.

## Results

- Added a new backlog item for selection-memory size limits and opportunistic pruning so context-state storage does not grow unbounded under higher chat volume.
- Added context-state helpers in `server.mjs`:
  - `clearSelection()`
  - `selectionMeta()`
- Extended `/api/brief/drilldown` responses with `contextMemory` metadata:
  - `usedRememberedSelection`
  - `before`
  - `after`
- Added `GET /api/brief/context?context=...` to inspect the currently remembered selection and TTL for a context key.
- Added `DELETE /api/brief/context?context=...` to clear remembered selection state for a context key.
- This gives the chat layer an explicit contract for inspecting, rotating, and clearing short-lived follow-up state instead of inferring behavior indirectly.

## Evidence
